`@ic-reactor/start` is the React/TanStack starter for **fully on-chain** ICP
frontends. It composes IC Reactor, [TanStack Router](https://tanstack.com/router),
and [`icp-cli`](https://github.com/dfinity/icp-cli) into a single Vite preset and
a project scaffold.

<Aside type="note" title="V0 scope">
  V0 is **CSR/static only**, deployed to an ICP asset canister. There is no SSR,
  no TanStack Start server functions, and no colocated backend functions.
  `@ic-reactor/start` is **not** a TanStack Start clone â€” it is the React/TanStack
  starter for ICP frontends. SSR and on-chain SEO rendering are future work (see
  the [Roadmap](#roadmap)).
</Aside>

## Create an app

```bash
pnpm create ic-reactor-start my-app
```

Or via the IC Reactor CLI:

```bash
ic-reactor create start my-app
```

Both write the same default V0 layout. Then:

```bash
cd my-app
pnpm install
icp network start -d   # start a local replica with Internet Identity
icp deploy             # deploy backend + frontend canisters
pnpm dev               # start Vite
```

<Aside type="tip">
  Canister IDs resolve at runtime through `icp-cli` and the `ic_env` cookie
  injected by [`@ic-reactor/vite-plugin`](/v4/packages/vite-plugin). There is
  **no `.env` file** for canister IDs.
</Aside>

## Generated layout

```text
my-app/
  icp.yaml              # icp-cli: backend + frontend canisters, managed local net
  ic-reactor.json       # Candid codegen config
  backend/
    canister.yaml       # Motoko recipe (V0 starter)
    main.mo             # counter sample canister
    backend.did
  frontend/
    canister.yaml       # Vite build + asset sync
    package.json
    vite.config.ts      # uses icReactorStart()
    index.html
    src/
      main.tsx          # TanStack Router + QueryClient bootstrap
      routeTree.gen.ts  # regenerated by the router plugin
      routes/__root.tsx
      routes/index.tsx  # backend query/update + II login sample
      lib/client.ts     # queryClient + clientManager
      lib/auth.ts       # AuthenticationManager + auth hooks
```

## The Vite preset

```ts
// frontend/vite.config.ts
import { defineConfig } from "vite"
import react from "@vitejs/plugin-react"
import { icReactorStart } from "@ic-reactor/start/plugin/vite"

export default defineConfig({
  plugins: [
    ...icReactorStart({
      canisters: {
        backend: { didFile: "../backend/backend.did" },
      },
    }),
    react(),
  ],
})
```

The preset:

- Passes normalized canister config into [`@ic-reactor/vite-plugin`](/v4/packages/vite-plugin)
  â€” the sole owner of Candid codegen and `ic_env` cookie injection.
- Includes TanStack Router plugin defaults (`target: "react"`, auto code-split).
- Proxies local `/api` to the `icp-cli` local gateway/replica.
- Honors `ICP_ENVIRONMENT`.
- Never hard-codes canister IDs.

### Options

| Option              | Type                               | Default            | Description                                            |
| :------------------ | :--------------------------------- | :----------------- | :----------------------------------------------------- |
| `canisters`         | `object \| CanisterConfig[]`       | â€”                  | Canister declarations (object keyed by name, or array) |
| `outDir`            | `string`                           | `src/declarations` | Output dir for generated declarations                  |
| `clientManagerPath` | `string`                           | `../../lib/client` | Import path to shared `clientManager`                  |
| `target`            | `"react" \| "core"`                | `"react"`          | Generated runtime target                               |
| `injectEnvironment` | `boolean`                          | `true`             | Inject `ic_env` cookie for local dev                   |
| `router`            | `boolean \| { autoCodeSplitting }` | `true`             | Include TanStack Router plugin                         |
| `proxyApi`          | `boolean`                          | `true`             | Proxy `/api` to local replica                          |

## How `ic_env` works

During `vite dev`, `@ic-reactor/vite-plugin` reads the local network state from
`icp` CLI (`icp network status`), resolves deployed canister IDs
(`icp canister status`), and sets an `ic_env` cookie containing the root key and
`PUBLIC_CANISTER_ID:<name>` entries. The `ClientManager` reads this cookie at
runtime, so the frontend never needs a baked-in canister ID.

## Deploy

Local:

```bash
icp deploy    # deploy backend + frontend canisters
icp sync      # sync frontend assets to the asset canister
```

Mainnet:

```bash
icp deploy -e ic
```

See the `icp-cli` [documentation](https://github.com/dfinity/icp-cli) for cycles,
controllers, and mainnet prerequisites.

## Why V0 is CSR/static

V0 optimizes for the fastest credible path to a working, fully on-chain app:

- The **asset canister** serves static files.
- The **browser** resolves canister IDs via the `ic_env` cookie.
- The **app** renders client-side.

This keeps the deploy model simple and auditable, and matches how the IC
asset canister works today.

### SSR, SSG, and on-chain SEO on ICP

- **SSR (server-side rendering)** â€” generic Node-style rendering â€” is **not** on
  the V0 roadmap. ICP does not run Node.
- **SSG / prerendering** of known public routes is planned for V0.2, producing
  static HTML still deployable to the asset canister.
- **On-chain SEO rendering** is the ICP-native long-term path: serving dynamic,
  certified HTML through a canister `http_request` entry point. This is tracked
  as V2 and is deliberately distinct from generic Node SSR.
